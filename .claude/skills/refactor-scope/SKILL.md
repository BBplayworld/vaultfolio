# 변경 범위 통제와 회귀 예측

이 스킬은 다음 두 패턴을 막는다:
1. **수정 범위가 옆 코드로 새어나가는 것** (예: 모바일만 고치려다 PC 깨짐, foreign 분기만 고치려다 domestic 영향)
2. **한 케이스를 고쳐 다른 케이스를 깨는 두더지잡기** (예: 381180 배당 빈도 고치니 QQQ/SPY 깨짐)

## 절대 규칙

1. **plan 모드에서 영향 케이스를 모두 열거하기 전에는 Edit/Write 금지.**
2. **"최소 변경"을 명시적으로 지킨다.** 요청된 분기·조건 밖의 코드는 손대지 않는다.
3. **사용한 적 없는 Tailwind/라이브러리 클래스는 사용 전 검증한다** (예: `size-4.5`는 존재하지 않음).

## 워크플로

### 1단계: 변경 의도와 단위 명확화

사용자 요청에서 다음을 추출한다:
- **무엇을** 바꿀 것인가 (한 줄)
- **어디만** 바꾸고 싶은가 (특정 컴포넌트/함수/분기/breakpoint)
- **무엇은** 바꾸면 안 되는가 (예: PC 레이아웃은 그대로)

불명확하면 사용자에게 묻는다.

### 2단계: 영향 표면 열거

해당 코드가 영향을 줄 수 있는 케이스를 **모두** 나열한다:

- **데이터 케이스**: 어떤 ticker/카테고리/통화/시점에서 분기되는가
  - 예: domestic vs foreign vs unlisted, KRW vs USD vs JPY, 장중 vs 장외, 영업일 vs 주말
- **UI 케이스**: 어떤 breakpoint·variant·상태에서 다르게 보이는가
  - 예: 모바일/PC, light/dark, 활성/비활성, 빈 상태
- **호출 케이스**: 이 함수를 호출하는 곳이 몇 군데인가 (Grep으로 확인)
- **저장 케이스**: localStorage 키, 캐시 상태, 공유 토큰 영향

### 3단계: 각 케이스의 예측 동작 기록

영향 케이스 각각에 대해 수정 후 동작을 한 줄로 예측한다:

```
| 케이스 | 수정 전 | 수정 후 예측 | 회귀 위험 |
|---|---|---|---|
| domestic | OK | OK (변경 없음) | 없음 |
| foreign  | NG | OK | 없음 |
| unlisted | OK | OK? (검증 필요) | 중간 — averagePrice 폴백 확인 |
```

예측이 모호한 케이스가 있으면 그 부분 코드를 먼저 읽어 검증한다.

### 4단계: 최소 변경 plan 제시

ExitPlanMode에 포함:
- 수정할 파일 목록과 정확한 변경 내용
- 단계 3의 영향 케이스 표
- "이 변경이 건드리지 않는 것" 명시 (예: "PC breakpoint는 변경 없음, foreign 분기는 변경 없음")
- 검증 방법 (어떤 입력으로 어떤 결과를 확인할 것인가)

### 5단계: 승인 후 좁게 수정

- 요청 분기 밖의 코드는 절대 손대지 않는다
- 비슷한 부분을 "기왕 하는 김에" 정리하지 않는다 (사용자가 명시적으로 요청한 경우 외)
- 새로 쓰는 클래스/유틸은 기존 코드베이스에 동일한 게 있는지 먼저 확인한다

### 6단계: 변경 후 자체 검증

- 수정한 모든 분기·breakpoint·케이스를 다시 확인
- 타입 체크 통과 여부 보고 (PostToolUse 훅이 자동 실행되지 않는 환경이면 수동 실행)
- 단계 3 예측표와 실제 결과를 대조 후 보고

## 안티패턴 — 하지 말 것

- 분기 A만 고친다 해놓고 분기 B의 동일 패턴도 "통일" 명목으로 같이 수정 (← 범위 누출)
- 한 케이스 고치고 "완료" 선언, 다른 케이스 영향 미확인 (← 두더지잡기 직전 상태)
- 모바일 클래스를 추가하면서 PC 동작 미검증 (← breakpoint 누락)
- 기존에 같은 역할 헬퍼가 있는데 새로 만듦 (← 중복 코드 생성)
- 사용한 적 없는 Tailwind 임의 클래스 사용 (← `size-4.5` 같은 비존재 클래스)

## 사용 예시

요청: "모바일에서 버튼 텍스트가 줄바꿈되게 해줘"
→ 단계 1: 어디만 = 모바일, 무엇은 = PC는 그대로
→ 단계 2: 영향 케이스 = 모바일·PC 두 breakpoint, 같은 버튼 컴포넌트의 다른 사용처들
→ 단계 3: PC는 그대로여야 함을 예측표에 명시
→ 단계 4: plan에 "sm: 이상은 변경 없음" 명시, 적용할 Tailwind 클래스 지정
→ 단계 5: 모바일 분기에만 클래스 추가, w-full 같은 광역 클래스는 회피
→ 단계 6: PC·모바일 둘 다 확인 후 보고

## 메타 규칙

- 이 스킬과 `/diagnose`는 자주 함께 쓰인다. 버그 픽스라면 먼저 `/diagnose`로 원인 확정 → 그 다음 `/refactor-scope`로 수정 범위 통제.
- "최소 변경"이 항상 최선은 아니지만, **기본값**은 최소 변경. 더 큰 정리가 필요하면 사용자에게 명시적으로 제안하고 승인을 받는다.
